# 규칙
- source에 실제로 나타난 내용만 사용한다.
- 존재하지 않는 수치, 성과, 아키텍처, 의사결정을 지어내지 않는다.
- 모르면 빈 배열로 둔다.
- 각 후보는 반드시 `source_refs`를 포함한다.
- `source_refs` 값은 아래 "# Raw Source" 섹션의 `### ` 헤더에 나타난 경로만 사용한다. 존재하지 않는 경로를 만들어내지 않는다.
- 한국어로 작성한다.
- 공식 영역(`20_Knowledge/`, `40_AgentMemory/Core/`, `30_Projects/*/Context.md`)을 직접 수정하는 문장을 쓰지 않는다. 후보로만 작성한다.
- 아래 "관련 기존 지식 노트" 목록에 있는 노트는 `[[stem]]` 형식의 wikilink로 body 안에서 참조한다.

# knowledge 후보 선별 기준

knowledge 배열에 넣으려면 다음 세 질문 **모두** "예"여야 한다.

1. **WHY가 있는가**: 코드나 git log를 읽으면 WHAT은 알 수 있다. 이 노트는 그 결정이 왜 그렇게 내려졌는지, 어떤 트레이드오프를 거쳤는지를 설명하는가?
2. **6개월 후에도 가치 있는가**: 구현 맥락을 잊은 뒤 다시 봐도 의사결정·설계·디버깅에 실질적 도움이 되는가?
3. **이 프로젝트를 넘어 적용 가능한가**: 비슷한 다른 시스템·상황에서도 쓸 수 있는 원칙·패턴·개념인가?

다음은 knowledge 후보가 **아니다** — 이 항목들이 source에 있어도 knowledge 배열에 넣지 않는다:
- "A 파일에 B 기능을 추가했다" — 코드를 보면 알 수 있는 구현 사실
- 커밋 메시지나 PR 변경 목록을 재서술한 것
- 필드 추가, 클래스명 변경, import 정리 같은 단순 코드 정리
- 공식 문서에 이미 있는 라이브러리·언어 기능 설명

knowledge가 없으면 `"knowledge": []`로 두는 것이 틀린 지식을 남기는 것보다 낫다.

# blog_idea 후보 선별 기준

blog_idea 배열에 넣으려면 다음 세 질문 **모두** "예"여야 한다.

1. **독자에게 실질적 가치가 있는가**: 이 글을 읽는 다른 개발자가 자신의 프로젝트에 적용할 수 있는 패턴·원칙·인사이트가 있는가?
2. **이 프로젝트를 모르는 사람도 읽을 수 있는가**: "WorkAgent를 개발하면서"가 아니라 "이런 상황에서 이런 접근이 효과적이었다"는 형태로 쓸 수 있는가?
3. **여러 작업 세션의 패턴을 종합하는가**: 단일 커밋·단일 기능이 아니라 반복되는 문제·해결 패턴·설계 결정을 묶어서 이야기할 수 있는가?

다음은 blog_idea 후보가 **아니다**:
- "X 기능 개발 후기" — 특정 기능 구현 과정 나열
- 단일 커밋이나 단일 버그 수정을 글감으로 삼은 것
- "우리 프로젝트에서는 이렇게 했다"로만 끝나는 것 (독자가 뭘 가져갈 수 있는지 불명확)

blog_idea가 없으면 `"blog_ideas": []`로 두는 것이 억지로 채우는 것보다 낫다.

# body 내 사실 주장 제약 (할루시네이션 방지)

body에 쓰는 모든 구체적 주장은 "# Raw Source" 텍스트에서 직접 확인 가능해야 한다.

- "~을 구현했다", "~를 해결했다"는 source에 해당 사실이 **명시적으로** 언급된 경우에만 쓴다.
- source에 없는 동작 방식, 성능 수치, 미구현 계획을 추정하거나 보완하지 않는다.
- source가 결과만 언급하고 과정을 생략했다면, 과정을 상상해서 채우지 않는다.
- 확인되지 않는 내용은 생략하거나 "source에서 확인 불가"로 명시한다.

# body 작성 기준
각 후보의 body는 **최소 200자 이상** Markdown으로 작성한다. 단순 요약이 아니라 나중에 다시 봐도 이해할 수 있는 실질적 내용을 담는다.

**knowledge body 구조:**
```
## 개념
핵심 개념을 2~4문장으로 설명한다.

## 왜 중요한가
실무에서 이 지식이 필요한 상황과 이유를 구체적으로 설명한다.

## 적용 방법 / 예시
실제 적용 방법이나 코드/명령어 예시를 포함한다.

## 관련 노트
[[관련노트stem]] — 연관 이유 한 줄
```

**decision body 구조:**
```
## 문제 상황
어떤 상황에서 결정이 필요했는지.

## 검토한 선택지
- 선택지 A: 장단점
- 선택지 B: 장단점

## 결정
최종 선택한 것.

## 근거
이 선택을 한 이유와 트레이드오프.

## 관련 노트
[[관련노트stem]] — 연관 이유 한 줄
```

**blog_idea body 구조:**
```
## 핵심 메시지
독자가 이 글에서 가져갈 수 있는 것을 한 문장으로. "X를 개발하면서 Y를 배웠다"가 아니라 "X 상황에서 Y 접근이 효과적인 이유".

## 독자 대상
이 글이 도움이 되는 사람. 프로젝트 외부의 독자를 상정한다.

## 목차 초안
1. 도입: 독자가 공감할 문제 상황 제시
2. 본론 1: 핵심 패턴/접근 설명
3. 본론 2: 실무에서 어떻게 적용하는지 (코드/예시 중심)
4. 결론: 독자가 내일 당장 적용할 수 있는 것

## 관련 노트
[[관련노트stem]] — 연관 이유 한 줄
```

**memory_patch body 구조:**
```
## 대상 파일
40_AgentMemory/XX_파일명.md

## 추가/수정할 내용
실제로 추가하거나 수정할 Markdown 텍스트.

## 이유
왜 이 내용을 AgentMemory에 반영해야 하는지.
```

# 출력 형식
아래 JSON 객체 하나만 출력한다. 코드펜스나 설명은 넣지 않는다.

      "body": "위 knowledge body 구조를 따른 상세 Markdown (200자 이상)",
      "project": "관련 프로젝트명 또는 빈 문자열",
      "tags": ["tag"],
